## SECTION B — Linux eBPF Agent

> nouveau crate : `arqenor-ebpf/` (C probes + Rust loader libbpf-rs)

- [x] **B1** — Créer le crate `arqenor-ebpf` (workspace member)
  - `arqenor-ebpf/src/probes/` — programmes eBPF en C (compilés avec clang)
  - `arqenor-ebpf/src/loader.rs` — loader Rust avec `libbpf-rs`
  - `arqenor-ebpf/src/events.rs` — struct `EbpfEvent` + ring buffer consumer

> **B2–B7 status (2026-04-26):** the 5 `.bpf.c` probes are now loaded + attached at runtime by
> `arqenor-ebpf/src/loader.rs::EbpfAgent::start()`. Skeletons are generated by `libbpf-cargo`
> through `build.rs`; each probe is opened, loaded, auto-attached via its `SEC()` annotations, and
> drained on a `tokio::spawn_blocking` task that forwards decoded `EbpfEvent`s on a bounded
> `mpsc::Receiver<EbpfEvent>`. Per-probe failures are demoted to `tracing::warn!` so a missing
> kprobe symbol on the running kernel does not take down the whole agent.
>
> **Not yet wired into `arqenor-core::pipeline::DetectionPipeline`** — the pipeline's input
> channels are typed (`ProcessEvent` / `FileEvent` / `ConnectionInfo`) and adding an
> `EbpfEvent` lane requires a follow-up refactor (own task, see `B7-followup`). The receiver is
> exposed publicly on `EbpfAgent::start()` so downstream consumers can drain it today.

- [x] **B2** — Probes exécution de processus `T1059`
  - `tracepoint/syscalls/sys_enter_execve` → capture filename + argv
  - `tracepoint/syscalls/sys_enter_execveat` → dirfd + filename

- [x] **B3** — Probes injection mémoire `T1055`
  - `kprobe/do_mmap` → flag PROT_EXEC | PROT_WRITE combinés
  - `kprobe/ptrace` → attach à un autre process (T1055.008)

- [x] **B4** — Probes persistence `T1574.006 / T1053`
  - `sys_enter_openat` → ouverture de `/etc/ld.so.preload`, `/etc/cron*`

- [x] **B5** — Probes privilege escalation `T1068`
  - `kprobe/commit_creds` → uid 0 escalation depuis process non-root

- [x] **B6** — Probe chargement de module kernel `T1014`
  - `kprobe/do_init_module` → nom du module via CO-RE

- [x] **B7** — Pipeline eBPF → arqenor-core (loader runtime)
  - `build.rs` génère les skeletons des 5 probes via `libbpf-cargo` 0.24.
  - `EbpfAgent::start()` retourne `(Self, mpsc::Receiver<EbpfEvent>)` ; chaque probe est
    open → load → auto-attach → ring-buffer drain dans un `spawn_blocking`.
  - Erreurs typées via `EbpfLoadError` (`SkelOpen` / `SkelLoad` / `Attach` / `RingBuffer`).
  - **Wiring pipeline** : ✅ — `arqenor-cli/src/commands/watch.rs::spawn_ebpf_bridge` boote
    l'agent côté Linux et bridge `EbpfEvent` → `Alert` via le canal existant `scan_tx`
    (pas de 5e lane dans `DetectionPipeline` — les events kernel passent par la même
    voie d'admission que les host scans Windows). `ProcessExec` est observability-only
    (pas d'alerte par event). Les 6 autres kinds mappent à `SENT-EBPF-*`.
